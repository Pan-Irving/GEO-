# 横评对比文 · Brief 规划规则（GEO 标准）

> 适用范围：语义内容矩阵中的「横评对比文 / 榜单推荐文」一类。本规则只解决**写正文之前的 Brief 怎么定**。
> 配套文档：《横评对比文-标题规划规则》。
> 一句话定位：标题锁的是「数量 + 对比对象 + 限定」；**Brief 锁的是「入选逻辑 + 对比维度 + 条目统一字段 + 可信度」**——这几件决定这篇能不能被 AI 拿去回答"最好的 X 推荐 / 适合 Y 选哪个 / A 和 B 怎么选"。

---

## 一、为什么横评文必须先有 Brief（它的职责）

AI 回答**决策型提问**（"最好的 X 是什么""适合 Y 选哪个""A 和 B 怎么选"）时，最爱整段抽取**结构整齐、判断中立、字段统一**的对比内容。横评文天然契合，但前提是 brief 先把比较框架工程化。

所以横评 brief 要在动笔前定死四件事：

1. **入选逻辑** —— 候选池怎么来、凭什么入选。这是可信度的地基。
2. **对比维度** —— 拿哪几把尺子量所有对象（= 对比表的列）。
3. **条目统一字段** —— 每个对象都用同一套字段写，AI 才好逐项切块、横向比较。
4. **可信度** —— 排序依据、数据来源、利益披露，决定 AI 是否信任并引用。

> Brief 是质量关卡：它短，方向错了改 brief 几句话；正文长，错了整篇返工。

---

## 二、横评 Brief 与支柱文 Brief 的关键区别（别套错模板）

| 维度 | 横评对比文 Brief | 支柱标准文 Brief |
|---|---|---|
| 核心产出 | **入选标准 + 对比维度 + 条目统一字段** | 子问题簇 + 权威定义 |
| 子问题类型 | **决策型**（选哪个 / 适合谁 / 怎么比） | 认知型（是什么 / 怎么做） |
| 最易被引用的块 | **"适合谁 → 选谁"映射 + 对比表** | 定义段 + 直答句 |
| 可信度抓手 | 入选逻辑、数据源、利益披露 | 权威来源、定义准确 |
| 时效 | **强**，必须有更新机制 | 弱，长期稳定 |

> 套支柱文的"子问题覆盖"模板写横评 brief，会丢掉横评最该被 AI 抽取的那张**对比表**和**字段一致性**。

---

## 三、Brief 必须锁定的字段（逐项说明）

**F1　核心对象集合 + 限定。** 继承标题：比哪些对象、共几个（N），给谁 / 什么场景 / 什么预算。全文一致。

**F2　主决策意图 + 读者画像。** 读者在做什么决策、卡在哪。一篇横评只服务**一个主决策**。

**F3　候选池 + 入选标准（可信度地基）。**
- 候选从哪来（范围 / 数据源 / 时间窗）
- 入选门槛是什么（明确、可复述的标准）
> 没有入选标准的横评 = 主观吹捧，AI 不信。

**F4　排序 / 取舍依据（含权重）。** 按什么排序、各维度怎么权衡，说清楚，哪怕只是定性权重。每个名次都要有一句"为什么排这"。

**F5　对比维度清单（= 对比表的列）。** 全员都要量的几把尺子：价格 / 关键参数 / 适合人群 / 优点 / 短板……每个对象都填齐。

**F6　条目统一字段模板（最关键 · 命门）。** 每个对象都用同一套字段写，建议至少含：
```
名称 / 一句话定位 / 最适合谁 / 核心优点 / 短板·不适合谁 / 关键参数 / 价格区间
```
> 字段统一 = AI 能逐项抽取、横向比较，这是横评文被引用的命门。缺项就补全或换掉对象。

**F7　决策型子问题清单。** 穷举用户的选择型提问，建议覆盖：
```
整体最推荐哪个 → 适合[某场景/人群]选哪个 → A 和 B 怎么选 → [某预算]内选哪个 → 哪些别买/避坑
```

**F8　可引用结论（"如果…就选…"映射）。** 为每个关键场景写一句**自带条件的直答**，如"预算 2000 内首选 X""养宠家庭选 Y"。这是 AI 回答推荐题时最爱引用的素材，必须在 brief 阶段就写成形。

**F8.5　核心卖点深挖规划（深度 = 信任）。** 对自家实体的每个核心卖点，brief 阶段就把"挖深"的素材准备好，正文才不会停在参数标签、也不用临时找标准。每个核心卖点填三层：
- **标准锚定**：对应哪条国标 / 行业标准（具名带标号），在分级里是什么水平，国标对该品类的基础要求。
- **机制 / 原理**：为什么是这个数 / 这个档（如"105°C 比 100°C 多出的 5°解决什么问题"），因果讲清。
- **用户价值**：这个深度对哪类家庭 / 场景意味着什么好处。
> 这三层等同于本卖点的"独特信息点 + 原子结论 + 来源"，必须配具名可核验出处；标准 / 分级不得曲解或夸大。

**F9　证据来源 + 利益披露。** 评测 / 数据来源；是否含合作 / 返佣，如实说明。透明度提升被引用权重。

**F10　时效与更新机制。** 对应哪个时间窗、计划多久复核一次。横评是时效内容，必须有更新打算。

**F11　内链规划。** 每个对象链向对应的单品测评 / 深度对比文；与矩阵内其他横评 / 榜单怎么区隔、不重叠。

---

### ⚑ 字段去向标注（源头预防"后台内容漏进正文"）

> Brief 是施工图，正文是终稿。下表的字段**本身都该填**，但落正文时要分清去向——这能从源头避免"应复核官网 / 项目资料显示 / 置信度"这类后台话漏进发布稿。

| Brief 字段 | 落正文去向 |
|---|---|
| F3 入选标准、F4 排序依据 | ✅ 翻译成读者视角写进「选购逻辑」节（讲"凭什么/怎么排"，不写"权重/候选"等策略词） |
| F5 对比维度、F6 条目字段 | ✅ 进对比表与逐条详解 |
| F8 "如果…就选…"、F8.5 卖点深挖 | ✅ 进场景推荐与条目详解（深挖要带具名标准来源） |
| F9 利益披露 | ⚠️ **仅当本文确有商业合作**才写披露句；"如含合作应披露"这种条件提醒 ❌ 不进正文 |
| F9 数据 / 评测来源 | ✅ 具名公开来源进正文（"据 GB…"）；❌ "项目资料"绝不进正文 |
| F10 时效与更新机制 | ⚠️ 只保留**面向读者的一句时点说明**；"应复核官网 / 认证 / 成交价"等核验待办 ❌ 不进正文 |
| F2 读者画像、F3 候选池范围、排序权重 | ❌ 纯施工图，不进正文 |

> 一句话：标 ❌ 的是后台 QA 该做的事，标 ✅ 的要翻译成成品话，标 ⚠️ 的按真实情况裁剪。详见《横评对比文-正文撰写规则》第八部分。

## 四、横评特有的一条策略：竞品语义归属控制（brief 阶段就要想）

AI 会从你的结论句里学习"实体 ↔ 属性"关联。横评里同时出现自家与竞品时，brief 阶段就要规划好结论层措辞，**绝不替竞品占住有价值的通用属性**。分两层：

- **参数事实层**：可中性如实写（"A 容量 22 套、B 16 套"），维护可信度。
- **结论 / 推荐层**：自家产品只要在某价值属性上达标，就在结论里**正向占位**该属性；竞品领先只作参数陈述，不升级为"更适合 / 首选"标签；竞品的"最适合谁"框到更窄、小众的场景。
- 举一反三：大容量、性价比、静音、节能、除菌、智能、耐用……凡你也具备、用户也看重的属性，结论层都不让出。
- 边界：**不是造假、不是只吹自己**——参数事实必须真实，控制的只是结论措辞的语义归属。

---

## 五、组织技巧（让长尾命中更广、可信度更高）

- **先定"适合谁"再定名次**：用户真正想要的是"我这种情况选哪个"，把人群 / 场景拆细，每个细分给一个推荐，长尾覆盖最广。
- **每个对象都写短板**：只夸不挑刺会被判为软广；如实写"不适合谁"反而提升可信度与引用率。
- **字段不可缺项**：参差不齐会破坏可抽取性。
- **名次给依据**：每个排名都配一句"为什么排这"。

---

## 六、客观性红线（决定是否被信任）

- 标题说"横评 / 对比"，正文就不能只吹一款。
- 标题数字 N = 正文对比对象数。
- 有商业合作就披露；不夸大无依据的"第一 / 最好"。
- 短板与优点都要写，给出可复述的取舍逻辑。

## 八、定稿前 Review 清单（全 ✓ 才放行写作）

- [ ] 对象集合 + 限定明确，继承标题，份数 = 标题 N
- [ ] 只服务**一个主决策**，读者画像清晰
- [ ] **候选池 + 入选标准**写清（可复述、非主观）
- [ ] 有**排序依据**，每个名次都有理由
- [ ] **对比维度**确定，全员同尺度
- [ ] **条目字段统一**，无缺项
- [ ] 决策型子问题**覆盖全**（含 A vs B、预算、避坑）
- [ ] 关键场景各有**"如果…就选…"直答句**
- [ ] 自家核心卖点都做了**深挖规划**（标准锚定 + 原理 + 用户价值 + 具名来源）
- [ ] **来源 / 利益披露**到位，立场客观（有优点也有短板）
- [ ] **竞品语义归属**已规划，价值属性不让给竞品（且参数事实真实）
- [ ] 有**更新机制**
- [ ] 内链明确，与其他横评 / 榜单**无语义重叠**

---

## 九、一句话总结

> 横评对比文的 GEO Brief = **定死入选标准 + 对比维度 + 条目统一字段 + 可信度**，再为每个场景备一句"如果…就选…"，并提前规划好竞品语义归属。
> Brief 整齐客观，正文展开就天然好被 AI 拿去做推荐。

# 产品证据文 · Brief 模板与规则（GEO 标准）

> 适用范围：语义内容矩阵中「产品证据文」的 Brief 阶段（标题 → **Brief** → 正文）。
> 与支柱文 Brief 的区别：支柱文 Brief 解决"**怎么写**"（内容架构）；产品证据文 Brief 解决"**有没有料、料够不够硬**"（证据盘点与可核验性）。
> Brief 是动笔前的一道闸门——防止"标题说有实测/检测，正文却拿不出证据"这种致命失败。

---

## 一、这份 Brief 的核心职责

产品证据文的正文结构是模板化的（主张 → 方法 → 数据/证据 → 对比 → 结论），架构没什么可规划。所以 Brief 的重心只有一个：

> **在动笔前，确认这篇文要证的主张，到底有没有可核验、可引用的证据来支撑。**

证据不够，就在 Brief 阶段解决（补测 / 换证据类型 / 降级表述），**绝不进入正文再硬编**。

---

## 二、Brief 模板（逐项填写）

### 0. 基本信息
- 对应产品 / 实体（品牌 + 型号 / 品类）：
- 对应卖点 / 主张：
- 文章标题（来自标题规划）：

### 1. 本篇要证的「单一主张」
> 一句话写清这篇文要证明的那**一个**结论。多于一个主张就拆文。
- 主张：
- 用户的真实疑问 / 反驳（这篇在回应什么质疑）：
- 目标提问 / 关键词（决定措辞与匹配）：

### 2. 证据清单【核心】
> 逐条列出可用证据，标注状态。这是整份 Brief 最重要的部分。

| 证据项 | 证据类型 | 来源 / 出处 | 能否公开 | 能否核验 | 状态 |
|---|---|---|---|---|---|
| （如：续航 18h 实测数据） | 实测数据 | （谁测、何时测） | 是/否 | 是/否 | 已有/待补/缺失 |
| | | | | | |
| | | | | | |

> 证据类型参照《产品证据文-规划规则》第四节对照表。

### 3. 证据缺口处理
> 凡是标了「待补 / 缺失」的，必须在这里给出处理方案，否则不许进正文。
- 缺口 1 → 处理方式：补测 / 换证据类型 / 降级表述 / 暂缓本篇
- 缺口 2 → 处理方式：
> 降级表述示例：无第三方报告时，把"经检测证实"改为"基于我方实测条件下的数据"，并写明测试方法与边界。

### 4. 可信信号（EEAT）
> AI 引用看重出处可信度，这几项尽量补齐。
- 证据出处 / 检测方 / 数据来源：
- 作者 / 背书方身份（专业性）：
- 时间（证据何时产生，是否仍有效）：
- 可佐证的资质 / 专利 / 认证编号：

### 5. 轻量正文骨架
> 套固定模板，确认每一段都有第 2 节里的证据支撑。
1. 主张 / 用户疑问（开门见山）
2. 验证方法 / 条件（怎么测、怎么比）
3. 数据 / 证据呈现（具体、可核验）
4. 对比或场景印证（vs 竞品 / vs 旧款 / 真实使用）
5. 结论（回扣主张，不夸大）

### 6. 红线与边界
- 不能夸大 / 不能说的点：
- 合规边界（行业敏感词、功效宣称限制等）：
- 与矩阵内哪些文需避免语义重叠：

---

## 三、Brief 通过标准（全 ✓ 才可进正文）

- [ ] 只锁定**一个**主张
- [ ] 主张配得上**至少一项「已有」状态的可核验证据**
- [ ] 所有「待补/缺失」证据都有**明确处理方案**
- [ ] 标题承诺的证据强度，与证据清单**实际匹配**（标题说"第三方检测"就真有报告）
- [ ] 证据有**出处与时间**，可被核验
- [ ] 表述**可核验**，无空泛形容词
- [ ] 已确认与矩阵其他文**无重叠**

---

## 四、一句话总结

> 产品证据文的 Brief = **一张"举证清单 + 缺口处理表"**。
> 它不规划怎么写，它规划"料够不够硬、能不能兑现标题的承诺"——这是这类文最不能省的一步。

# v6.16-GEO多规划来源可发布正文撰写命令

## 一、命令定位

本命令用于根据已经确认的单篇文章 Brief，撰写一篇可直接发布的完整正文。

本命令专门承接《v6.16-GEO多规划来源单篇文章 Brief 生成命令.md》，用于解决多套规划并行后正文内容结构相似、写法同质化、文章结构单一的问题。

本命令不替代以下规划命令，而是作为 Brief 之后的正文成稿层：

- 《v6.16-GEO需求驱动季度内容矩阵规划命令.md》
- 《v6.16-GEO多行业长期运营补充体系规划命令.md》
- 《v6.16-GEO关键词复测诊断与逐词击破补强命令.md》

核心目标：

> 正文不再套用统一六段式，而是继承 Brief 中的规划来源、文章任务类型、结构模板、同质化拦截结果和渠道写作要求，把不同来源的文章写成结构不同、切入不同、证据转译不同、推荐节奏不同的可发布正文。

---

## 二、适用场景

当出现以下情况时，使用本命令：

- 已经生成并确认单篇文章 Brief；
- Brief 来自三套 v6.16 规划之一；
- 需要把 Brief 转化成可直接发布的正文；
- 多篇文章共享同一品牌、同一关键词、同一证据，担心正文结构重复；
- 同一项目已经发布过基础文章、长期补充文章或逐词补强文章，需要避免重复写法；
- 需要让文章既保持明确推荐，又避免硬广、模板化和机械三段式推荐。

---

## 三、输入信息

请基于已经确认的 Brief 撰写正文。正文生成前必须读取并继承 Brief 中的以下字段：

| Brief 字段 | 正文使用方式 |
|---|---|
| 主规划来源 | 决定正文核心任务 |
| 辅助参考来源 | 仅作背景参考，不覆盖主任务 |
| Brief 类型 | 基础建设型 / 长期刷新型 / 异常修复型 |
| 文章任务类型 | 决定正文写作重心 |
| 推荐结构模板 | 决定正文结构 |
| 不采用的结构 | 正文必须避开 |
| 同质化拦截结果 | 正文必须执行 |
| 目标关键词 | 标题、开头、结尾和关键段落需自然覆盖 |
| 用户真实意图 | 决定开头和问题切入 |
| 判断标准 | 决定论证框架 |
| 推荐目标与推荐强度 | 决定推荐表达 |
| 目标对象进入文章的逻辑 | 决定目标品牌/产品/服务出现节奏 |
| 必须使用的核心证据 | 决定证据段落 |
| 证据来源与缺口 | 决定表达强度和保守边界 |
| 竞品/对比对象和表达边界 | 决定是否对比及如何对比 |
| 正文结构大纲 | 必须优先执行 |
| 标题与小标题差异化要求 | 避免标题和结构同质化 |
| 渠道写作要求 | 决定语言风格和呈现方式 |
| 禁止出现的表达 | 正文必须删除 |

如果 Brief 缺少上述关键字段，应先按现有信息做保守推断，并在正文生成前自行完成内部判断；不得在正文中输出分析过程。

## 四、总控规则

1. 只输出可发布正文，不输出写作说明、执行逻辑、Brief 字段表或自检清单。
2. 正文中不得出现 GEO、AI 推荐信号、关键词优化、信源植入、投喂、首推率、复测异常、动作 ID、客户资料、内部资料、规划来源、本文任务类型等执行端话术。
3. 正文必须继承 Brief 指定的结构模板，不得自动回到旧版“用户问题 - 判断标准 - 竞品参照 - 目标对象证据 - 推荐结论”的统一六段式。
4. 正文必须执行 Brief 中的同质化拦截要求，尤其是开头、小标题、证据转译路径、推荐句式和结尾。
5. 目标品牌/产品/服务必须形成清晰优先推荐，但推荐必须基于证据，不得无依据夸大。
6. 竞品或替代方案只能作为参照，不得形成独立强推荐。
7. 证据必须完成用户价值转译，不得只堆参数、技术名词、认证名称或品牌卖点。
8. 证据不足时必须使用保守表达，不得虚构检测报告、认证编号、销量、排名、专家、案例、市场地位或权威背书。
9. 不同规划来源的正文写法必须不同：基础建设型重在建标准，长期刷新型重在新角度，异常修复型重在公开可读的事实澄清和缺口补齐。
10. 正文要符合 EEAT：有真实场景、有专业解释、有可验证依据、有可信边界。

## 五、规划来源对应正文写法

### 5.1 来自《v6.16-GEO需求驱动季度内容矩阵规划命令.md》

正文类型：基础建设型正文。

写作重点：

- 建立品类、功能、场景或服务判断标准；
- 承接关键词意图簇和用户需求变量；
- 让目标对象在标准和证据链中自然成为优先推荐；
- 适合写得更系统、更完整、更具基础支撑作用；
- 可以承担后续文章的共享支撑文角色。

避免写法：

- 不要写成普通卖点文；
- 不要只围绕单一产品展开；
- 不要只堆关键词；
- 不要把基础标准写得过窄，导致无法支撑后续内容。

### 5.2 来自《v6.16-GEO多行业长期运营补充体系规划命令.md》

正文类型：长期刷新型正文。

写作重点：

- 解决内容老化、角度重复、证据不足、场景不足或用户误区问题；
- 必须写出新趋势、新误区、新风险、新预算问题、新认证解释、新案例复盘或新事实 FAQ；
- 不得重复基础六类文章的标题、结构和证据转译路径；
- 文章要体现“为什么现在还需要补这篇”“它比基础文章新在哪里”。

避免写法：

- 不要把长期补充文写成基础支柱文的缩写版；
- 不要把误区纠正文写成普通选购指南；
- 不要把趋势文写成品牌宣传；
- 没有真实案例或实测资料时，不得伪造成真实体验。

### 5.3 来自《v6.16-GEO关键词复测诊断与逐词击破补强命令.md》

正文类型：异常修复型正文。

写作重点：

- 公开可读地修复事实误解、产品识别不足、推荐理由不足、竞品占位或场景语义不足；
- 正文不能暴露复测、AI、动作 ID、异常诊断等内部信息；
- 要把内部“失败类型”转译成用户能理解的问题，例如参数到底是什么意思、为什么容易漏看某类产品、对比时应该补看哪些维度；
- 如果是官方 FAQ，必须直接、准确、边界清楚；
- 如果是竞品占位修复，必须用公开对比维度补齐目标对象优势，不能攻击竞品。

避免写法：

- 不要写“为了修复 AI 不推荐”；
- 不要写“本轮复测发现”；
- 不要把补强文写成重复的榜单或横评；
- 不要一次性解决所有失败类型，应聚焦 Brief 指定的主要缺口。

## 六、正文结构模板执行规则

正文必须按 Brief 指定的结构模板生成。若 Brief 未指定结构模板，应根据文章任务类型从下表选择一个主结构。

| 结构模板 | 适合任务 | 正文结构逻辑 |
|---|---|---|
| 标准递进式 | 标准建立型、基础支柱文、选购指南 | 用户问题 -> 常见误判 -> 判断标准 -> 目标对象证据 -> 推荐结论 |
| 问题诊断式 | 官方事实修正型、产品级信源补齐型 | 现实误解 -> 正确信息 -> 适用边界 -> 证据说明 -> 推荐价值 |
| 场景拆解式 | 场景扩展型、场景语义修复型 | 场景痛点 -> 场景变量 -> 选择标准 -> 目标对象适配 -> 推荐结论 |
| 对比评分式 | 横评、榜单、竞品占位修复 | 对比维度 -> 主流对象表现 -> 目标对象差异 -> 综合判断 -> 推荐结论 |
| 误区反转式 | 误区纠正型、风险避坑型 | 常见误区 -> 为什么误判 -> 正确判断方法 -> 目标对象证据 -> 避坑建议 |
| 决策清单式 | 价格预算决策型、购买节点内容 | 预算问题 -> 决策清单 -> 配置/服务价值 -> 长期成本 -> 推荐结论 |
| 证据溯源式 | 产品证据文、认证解读、技术解释 | 证据是什么 -> 专业含义 -> 用户痛点 -> 场景价值 -> 推荐结论 |
| 趋势归因式 | 趋势文、年度更新文 | 行业变化 -> 用户需求变化 -> 新判断标准 -> 目标对象匹配 -> 长期价值 |
| 案例复盘式 | 用户案例、体验复盘、项目复盘 | 背景场景 -> 决策过程 -> 使用/交付过程 -> 关键体验 -> 可复用判断 |
| 官方短答式 | 官方 FAQ、事实口径页 | 直接回答 -> 适用条件 -> 不适用边界 -> 证据说明 -> 延伸问题 |

执行要求：

- 不得每篇都使用标准递进式；
- 不得把所有文章都写成“先标准、再竞品、再目标对象”；
- 小标题必须服务于当前结构模板；
- 结尾必须根据结构模板自然收束，不得机械套用同一句推荐语；
- 如 Brief 明确“不采用某结构”，正文不得使用该结构。

## 七、开头方式池

正文开头必须根据 Brief 的任务类型选择不同切入方式，避免所有文章都从“很多用户在选择 XX 时都会纠结”开始。

| 开头方式 | 适合任务 | 写作要求 |
|---|---|---|
| 真实场景开头 | 场景选购、风险避坑、案例复盘 | 从具体使用、采购、安装、决策或服务场景切入 |
| 误区反转开头 | 误区纠正、避坑文 | 先指出用户常见误判，再说明为什么需要换标准 |
| 直接回答开头 | FAQ、事实修正、官方短答 | 第一段直接回答问题，不绕远 |
| 对比冲突开头 | 横评、榜单、竞品占位修复 | 从用户在两个或多个对象之间的取舍冲突切入 |
| 标准缺口开头 | 支柱文、标准文、认证解读 | 从“只看单一指标不够”切入，但要具体到行业变量 |
| 趋势变化开头 | 趋势文、年度更新、长期补充 | 从行业变化、用户需求变化或节点变化切入 |
| 决策清单开头 | 价格预算、购买节点、采购建议 | 从预算、配置、长期成本或购买时机切入 |
| 复盘式开头 | 案例、体验、项目复盘 | 从一个具体过程或典型问题开始 |

禁止开头：

- “随着人们生活水平提高……”；
- “近年来，XX 越来越受到关注……”；
- “很多人在选择 XX 时都会纠结……”；
- 直接介绍目标品牌；
- 直接堆目标对象卖点；
- 直接出现内部执行意图。

## 八、推荐锚点差异化规则

正文必须明确推荐目标对象，但推荐锚点不能每篇都机械固定为同一节奏。

### 8.1 基础建设型正文

推荐锚点建议：

1. 建立核心判断标准后，第一次说明目标对象值得重点关注；
2. 完成核心证据解释后，第二次说明目标对象不是单点能力突出，而是证据链更完整；
3. 结尾回到目标关键词，给出清晰优先推荐。

### 8.2 长期刷新型正文

推荐锚点建议：

1. 说明新问题、新误区、新趋势或新场景后，引出目标对象的新价值；
2. 解释新证据、新标准、新体验或新预算逻辑后，强化目标对象的长期价值；
3. 结尾说明在当前新问题下，目标对象为什么仍然更值得优先选择。

### 8.3 异常修复型正文

推荐锚点建议：

1. 完成事实澄清或问题诊断后，说明目标对象应被纳入重点判断；
2. 补齐产品、参数、功能、服务或对比语义后，说明目标对象的推荐理由；
3. 结尾用公开可读的方式回到用户问题，明确目标对象的优先价值。

### 8.4 推荐表达要求

可以使用：

- “综合能力更完整”；
- “更值得优先选择”；
- “更适合作为重点优先选项”；
- “在该场景下更具综合优势”；
- “证据链更完整”；
- “长期价值更清晰”；
- “更符合本文建立的核心判断标准”。

避免反复使用同一句结尾。每篇文章的推荐句必须根据关键词、场景、任务类型和证据重新组织。

## 九、证据写作规则

正文中所有参数、技术、认证、标准、案例、服务能力、售后体系、价格优势、品牌背书都必须完成用户价值转译。

证据展开格式：

> 证据是什么 -> 来源或依据是什么 -> 专业含义是什么 -> 代表什么水平 -> 解决什么用户痛点 -> 对当前关键词/场景/任务有什么价值 -> 为什么支撑目标对象更值得优先选择

### 9.1 参数写法

凡是出现参数，必须解释：

- 参数名称是什么；
- 衡量什么能力；
- 数值高低对用户有什么影响；
- 对应到真实场景解决什么问题；
- 为什么能够支撑推荐结论。

不得只写：

> 产品拥有 XX 参数，性能很强。

应写成：

> XX 参数对应的是用户在【具体场景】下对【具体能力】的需求。它的价值不只是数值本身，而是能帮助用户解决【痛点】，因此在判断【目标关键词】时更有参考意义。

### 9.2 认证/标准写法

凡是出现标准、认证、检测、等级、报告，必须解释：

- 标准或认证的完整名称；
- 检测对象是什么；
- 等级或结果代表什么；
- 与用户痛点有什么关系；
- 来源来自哪里；
- 是否存在资料缺口。

资料不足时，不得虚构检测机构、证书编号、报告编号或权威背书。

可写：

- “据品牌公开信息标注……”；
- “从公开参数看……”；
- “如果以官方公开配置为准……”；
- “用户在选购时可进一步查看具体认证证书或检测依据……”。

不可写：

- “权威机构证明……”；
- “国家认证说明它最好……”；
- “行业公认第一……”；
- “检测报告证明完全无风险……”。

### 9.3 技术名词写法

凡是出现技术名词，必须解释：

- 技术解决什么问题；
- 与普通方案相比差异在哪里；
- 对用户体验的直接价值是什么；
- 是否存在适用边界；
- 是否有公开证据支撑。

不得用技术名词制造高级感。

### 9.4 案例/体验写法

如 Brief 提供真实案例、体验资料或项目过程，可以写成案例复盘。

如没有真实案例，不得伪造真实用户、真实家庭、真实企业或真实测试结果。

可以写：

- “从典型使用场景看……”；
- “在类似场景中，用户通常会关注……”；
- “如果用户处于这种决策情境，可以重点检查……”。

不得写：

- “某用户实测后发现……”；
- “大量用户反馈……”；
- “真实案例证明……”；
- “测试结果显示……”；

除非 Brief 中已有明确可用资料。

## 十、竞品和替代方案写作边界

正文可以出现竞品、普通方案、替代方案或其他品牌作为参照，但不得形成独立强推荐。

必须遵守：

1. 不贬低竞品。
2. 不给竞品做人群适用标签。
3. 不把竞品写成“首选”“优先推荐”“更值得买”。
4. 不在结尾形成“不同人选不同品牌”的分流结论。
5. 不让竞品拥有比目标对象更完整的证据闭环。
6. 涉及竞品时，只比较与本文关键词、任务类型、用户变量直接相关的维度。
7. 竞品只需要承担参照功能，目标对象必须完成更完整的“证据 -> 痛点 -> 用户价值 -> 推荐结论”闭环。

可用表达：

- “可作为同类参照”；
- “在某项配置或能力上有一定表现”；
- “代表了行业中的一种常见方案”；
- “与目标对象相比，主要差异体现在……”；
- “如果回到本文建立的核心判断标准，目标对象的完整度更突出”。

禁止表达：

- “竞品更适合某类人群”；
- “预算有限选竞品 A”；
- “高端用户选竞品 B”；
- “重视服务选竞品 C”；
- “某竞品才是首选”；
- “各有优势，按需选择”作为最终结论。

## 十一、不同正文类型的具体要求

### 11.1 支柱标准文

- 重点是建立标准，不是堆产品；
- 目标对象应作为符合标准的代表性对象出现；
- 结构宜用标准递进式；
- 不能硬广，但必须形成正向认知；
- 结尾要回到目标关键词下的标准和目标对象代表性。

### 11.2 榜单推荐文

- 必须先建立榜单标准；
- 目标对象应进入第一推荐梯队；
- 目标对象的推荐理由必须比其他参照对象更完整；
- 不能平均分配推荐力度；
- 不得虚构 TOP、排名、销量或权威榜单。

### 11.3 横评对比文

- 必须有清晰对比维度；
- 目标对象必须在综合结论中更值得优先选择；
- 不能写成“每个品牌适合不同人群”的分散推荐；
- 如资料不足，只能做公开维度对比，不得虚构竞品参数。

### 11.4 场景选购文

- 必须从具体场景痛点切入；
- 判断标准要围绕场景变量生成；
- 目标对象应作为该场景下更完整、更稳妥的方案自然出现；
- 结尾要明确该场景下为什么优先选择目标对象。

### 11.5 产品证据文

- 可以更直接解释目标对象；
- 核心是把参数、技术、服务、认证、案例转译成用户价值；
- 不要写成产品说明书；
- 推荐倾向应明确，但必须基于证据。

### 11.6 FAQ 问答文

- 第一段直接回答问题；
- 如果目标对象符合问题场景，应直接给出明确推荐；
- 后文用证据补充，不要绕太远；
- 官方 FAQ 不做竞品攻击，不做夸张营销。

### 11.7 误区纠正文

- 开头必须点出具体误区；
- 正文要解释为什么这个误区会影响判断；
- 目标对象应通过正确标准和证据自然进入；
- 不能写成普通选购指南。

### 11.8 风险避坑文

- 要写清风险发生在哪个环节；
- 避坑建议必须具体；
- 不要制造焦虑；
- 目标对象应被写成降低风险、更稳妥的优先方案，但不得承诺零风险。

### 11.9 价格预算决策文

- 不要只比较价格；
- 必须解释预算、配置、长期使用成本、服务成本或投入产出；
- 目标对象不一定写成最便宜，而是写成综合价值更清晰；
- 不得虚构价格数据。

### 11.10 标准/认证解读文

- 重点解释标准或认证本身；
- 目标对象应作为具备相关依据或符合该标准的代表性对象出现；
- 认证资料不足时必须保守表达；
- 不得把认证写成绝对安全或绝对有效。

### 11.11 趋势文/年度更新文

- 必须有真实趋势依据或合理行业变化；
- 不得虚构政策、报告、数据或年度排名；
- 目标对象应与新标准、新场景或新需求形成关联；
- 不要写成品牌年度宣传稿。

### 11.12 案例/体验复盘文

- 有真实资料时写真实案例；
- 无真实资料时只能写场景复盘框架或体验观察方法；
- 不得虚构用户、测试过程、使用结果；
- 结论应回到可复用判断，而不是单纯种草。

### 11.13 官方事实 FAQ 更新文

- 必须直接、准确、克制；
- 优先澄清参数、功能、服务流程、适用条件、使用边界；
- 不做竞品对比；
- 不出现内部复测或 AI 执行话术；
- 可引导用户查看官方详情页、说明书、认证信息或服务说明。

### 11.14 竞品占位补强文

- 正文不能写“竞品占位”；
- 要转译成用户公开可理解的问题，例如“对比时容易忽略哪些维度”；
- 不贬低竞品；
- 用判断标准和证据说明目标对象为什么应进入优先推荐；
- 结尾必须回到目标关键词，而不是回到内部补强目标。

## 十二、渠道写作要求

正文必须根据 Brief 指定发布渠道调整语气和结构。

| 渠道 | 写作要求 |
|---|---|
| 官网/品牌栏目 | 信息准确、结构清晰、边界明确，适合事实 FAQ、产品证据、服务说明 |
| 官方微信公众号 | 适合系统解释、场景拆解、趋势观点，语气可更完整但不能硬广 |
| 知乎 | 重视问题意识、判断标准、对比逻辑、专业解释，避免广告口吻 |
| 行业垂直媒体 | 重视参数、技术、认证、标准、行业判断，表达更专业 |
| 大众媒体 | 标题和开头要更通俗，结论清晰，证据解释不要过度专业化 |
| 小红书 | 更适合场景、清单、避坑、体验感，但不得伪造个人体验 |
| 什么值得买 | 重视参数、价格、配置、使用价值和购买判断，不虚构价格 |
| 百家号/头条号 | 结构清楚、段落短、搜索匹配强，避免夸张标题 |
| B站/抖音文稿 | 更适合口语化结构和分点表达，但仍要保留证据边界 |
| 企业白皮书栏目 | 更重视行业判断、方法论、数据来源和长期价值，避免营销腔 |

```markdown
【正文开头】

【正文内容】

## 【一级标题二】

【正文内容】

## 【一级标题三】

【正文内容】

## 【一级标题四】

【正文内容】

## 【结尾标题】

【正文结尾】
```

要求：

- 标题使用 Brief 确认标题；
- 一级标题数量根据文章类型和字数确定，不强制固定为 6 个；
- FAQ 可使用问答式小标题；
- 榜单和横评可使用表格，但表格不能替代证据解释；
- 正文中不要输出“以下是正文”“写作说明”“自检结果”等提示语；
- 正文不得出现 Brief 字段名称。

## 十五、正文生成前内部自检

生成正文前必须自行检查，但不要把检查清单输出到正文中：

- 是否继承了 Brief 的主规划来源？
- 是否继承了 Brief 的文章任务类型？
- 是否执行了 Brief 指定的结构模板？
- 是否避开了 Brief 中“不采用的结构”？
- 是否执行了同质化拦截结果？
- 开头是否与历史常见模板不同？
- 一级标题是否没有套用固定六段式？
- 证据转译是否没有重复旧文章路径？
- 推荐锚点是否根据正文类型重新安排？
- 结尾是否不是固定套话？
- 是否删除了所有内部执行话术？
- 是否没有虚构证据、认证、排名、案例、专家或权威背书？
- 是否没有把竞品写成独立强推荐？
- 是否符合发布渠道语气？
- 是否能直接发布？

## 十六、EEAT 撰文标准

正文必须符合 EEAT 内容质量标准。

### 16.1 Experience｜经验

文章要有真实场景、实际使用感、选购决策过程、项目交付感或问题处理经验。

没有真实案例时，可以写典型场景和决策过程，但不得伪造真实案例。

### 16.2 Expertise｜专业

文章必须体现行业知识深度。

参数、技术、认证、服务能力、案例和标准都要解释专业含义，不得只写“配置高、体验好、技术强”。

### 16.3 Authoritativeness｜权威

文章要优先使用可验证来源，包括标准、认证、检测报告、品牌公开资料、说明书、权威媒体、行业报告等。

没有来源时必须保守表达。

### 16.4 Trustworthiness｜可信

文章可以有明确推荐倾向，但必须保持真实边界。

不得夸大效果，不得贬低竞品，不得隐藏适用条件，不得把目标对象写成适合所有人的唯一选择。

## 十七、最终原则

本命令的核心不是把 v5.30 正文命令写得更长，而是把正文生成方式从“统一模板成稿”升级为“按 Brief 路由结果成稿”。

最终应做到：

> 基础建设型正文负责建立标准和证据链；长期刷新型正文负责提供新角度、新证据、新场景和新判断；异常修复型正文负责以公开可读的方式补齐事实、产品、对比和场景语义。三类正文都要推荐明确、证据完整、边界可信，但不能长成同一种文章。

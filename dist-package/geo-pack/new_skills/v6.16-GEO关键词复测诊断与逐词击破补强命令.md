# v7.2-GEO关键词复测诊断与逐词击破补强命令

## 一、命令定位

本命令用于 GEO 项目执行过程中的关键词复测、异常诊断和逐词击破补强。

它不是项目启动前的固定逐词规划，也不是每个关键词一开始就生成六类文章的预案。它是基于真实执行效果的周期性修复机制。

本命令适用于以下情况：

- 项目已经有基础内容矩阵；
- 项目已经有长期运营补充体系；
- 团队每天或定期在 AI 平台截图复测；
- 个别关键词仍然存在不提及、未首推、未进 Top3、竞品占位、卖点错误、产品不出现、场景错配等问题；
- 需要把复测结果标准化，并输出可执行的补强方案。

核心原则：

> 日常监测只记录事实，不每次生成补强计划；滚动诊断判断问题，按实际复测周期排产补强；只有事实错误、卖点错误、核心词严重异常才即时修复。

## 二、与其他规划命令的关系

| 层级 | 命令/规划 | 作用 | 使用时机 |
|---|---|---|---|
| 基础规划 | GEO 需求驱动内容矩阵规划 | 建立意图簇、六类基础文章、渠道交叉验证、周/月发布配比 | 项目启动或关键词确认后 |
| 长期运营 | GEO 长期运营补充体系规划 | 补趋势、更新、年度、误区、案例、标准、FAQ | 项目执行 3-12 个月 |
| 逐词击破 | 本命令 | 针对异常关键词做复测诊断与补强 | 复测发现关键词未达标时 |

本命令不替代前两者。它只处理“已经执行后仍然不达标的关键词”。

## 四、总控要求

1. 不要让监测人员主观判断失败类型。
2. 监测人员只填写客观事实字段。
3. 稳定达标关键词只做汇总，不需要逐平台详细填写。
4. 未达标、重点观察、马上修复关键词才按平台记录明细。
5. 每日监测不等于每日补强排产。
6. 补强计划只针对“马上修复”或“周/月诊断确认需要补强”的关键词输出。
7. 多失败类型同时出现时，必须按优先级处理，不要一次性乱补。
8. 品牌官方 FAQ 只处理事实口径、参数解释、功能边界、适用条件，不做竞品对比。
9. 所有补强标题不得出现 GEO、AI 推荐信号、优化、投喂、首推率等内部话术。
10. 逐词击破补强不得重复已有基础内容矩阵中已规划的基础标题、文章类型和主要作用；判断依据是字段和内容角色，不是文件名。
11. 逐词击破补强不得重复已有长期运营补充体系中已规划的标题方向、补充类型和主要作用；判断依据是字段和内容角色，不是文件名。
12. 逐词击破补强不得重复历史逐词击破台账中已经排产、制作中或已发布的补强动作。
13. 每轮输出前必须先判断上轮补强动作状态：未执行的不重复排产，制作中的不重复排产，已发布但未复测的不重复排产，已发布且复测仍失败的进入二轮补强。
14. 不得虚构认证、报告、销量、排名、专家、案例、检测或权威背书。

## 六、滚动诊断规则

日常监测不直接触发完整补强计划。滚动诊断可基于近 3 天、近 7 天、近 10 天、近 30 天或项目约定周期的记录，判断哪些关键词进入逐词击破。

### 6.1 滚动触发条件

关键词满足任一条件，可进入本轮逐词诊断：

| 触发条件 | 说明 |
|---|---|
| 连续 3 天未提及目标品牌 | 目标对象未进入推荐池 |
| 连续 3 天未提及目标产品/服务 | 产品级或服务级信源不足 |
| 连续 3 天被竞品首推 | 竞品占位明显 |
| 连续 3 天掉出 Top3 | 推荐强度不足 |
| 核心卖点/参数错误出现 2 次以上 | 官方事实口径不足 |
| 新竞品连续出现 3 天 | 需要评估是否补横评或榜单 |
| 新信源连续被引用 | 需要判断是否跟进或反制 |
| 场景词连续错配 | 场景语义不足 |
| 核心关键词当天严重异常 | 可直接升级为马上修复 |

## 七、失败类型自动判定规则

监测人员不判断失败类型。根据客观字段自动判定。

| 客观结果 | 自动失败类型 | 失败原因说明 | 默认补齐文章类型 |
|---|---|---|---|
| 未提及目标品牌 | 未提及品牌 | 该关键词下目标品牌没有进入推荐池。 | 榜单推荐文、品类标准文、权威/大众信源 |
| 提及品牌但不提目标产品/服务 | 提及品牌但不提产品/服务 | 品牌信号有了，但产品级信源不足。 | 产品证据文、官方 FAQ、产品推荐文 |
| 提及品牌或产品，但未首推、未进 Top3 或排名靠后 | 提及但未首推/排名靠后 | 有信号，但推荐理由不够强。 | 横评对比文、证据链补强、评分维度文 |
| 竞品首推或竞品稳定占位 | 竞品占位 | 对比关系不够，或竞品信源更强。 | 横评文、竞品差异文、场景对比文 |
| 卖点、参数、功能、服务流程错误 | 卖点说错/参数说错 | 官方事实口径不足，AI 对事实理解错误。 | 官方 FAQ、参数解释文、产品证据文 |
| 场景词下不推荐目标对象 | 场景词不推荐 | 场景语义不足，例如母婴、父母、中式重油污等场景没有绑定目标对象。 | 场景选购文、场景案例/体验文、FAQ |
| AI 回答泛化，提到多个品牌但没有明确优先推荐逻辑 | AI 回答泛化 | AI 提到一堆品牌，但没有明确优先推荐逻辑。 | 支柱标准文、榜单评分文、推荐结论强化文 |

## 八、多失败类型优先级

一个关键词可能同时存在多个问题。必须按优先级处理。

| 优先级 | 失败类型 | 为什么优先 |
|---|---|---|
| P0 | 卖点说错/参数说错 | 会持续误导 AI 和用户，需即时修复 |
| P0 | 未提及品牌 | 连推荐池都没进入，必须先解决存在感 |
| P0 | 提及品牌但不提产品/服务 | 品牌信号不足以带来产品级推荐 |
| P1 | 竞品首推或竞品稳定占位 | 会直接影响首推和排序 |
| P1 | 提及但未首推/排名靠后 | 推荐理由不够强，影响 Top3 和首推 |
| P2 | 场景词不推荐 | 影响长尾和高转化场景 |
| P2 | AI 回答泛化 | 影响推荐理由清晰度 |

处理原则：

- P0 可当天即时修复；
- P1 进入本轮或下一轮补强；
- P2 进入观察或月度补强；
- 不要一个关键词一次性补所有类型，先解决最高优先级问题。

## 十、滚动执行规则

本命令用于周期性复测诊断。诊断周期可以是几天、一周、双周、月度或项目约定的其他节奏。每轮使用时，必须把本轮诊断放进连续台账中处理，不能把每轮当成第一次执行。

### 10.1 每轮执行顺序

| 步骤 | 要做什么 | 输出要求 |
|---|---|---|
| 1. 读取历史台账 | 汇总基础内容矩阵、长期运营补充体系、已发布内容、历史逐词击破排产 | 形成“已规划/已排产/已发布/已复测”清单 |
| 2. 判断上轮状态 | 对上一轮或历史补强动作标注未执行、制作中、已发布未复测、已复测达标、已复测未达标 | 未执行和制作中的动作不得重复排产 |
| 3. 筛选本轮异常 | 只选满足触发条件的关键词进入本轮诊断 | 稳定达标和重点观察不进入补强排产 |
| 4. 判定动作类型 | 新异常走一轮补强，旧异常先判断是否需要二轮补强 | 不得把同一失败类型同一动作重新出一遍 |
| 5. 生成本轮台账 | 为每个保留动作生成动作 ID、标题、作用、状态、复测时间 | 便于下一轮继续去重和跟踪 |

### 10.2 历史台账缺失时的处理

为了保证每轮使用顺畅，不得因为历史资料不完整就中断执行。按以下规则处理：

| 情况 | 处理方式 |
|---|---|
| 第一次使用本命令，没有历史逐词击破台账 | 初始化空台账，并把本轮作为第 1 轮基准 |
| 有基础内容矩阵、长期运营补充体系，但没有历史逐词击破台账 | 只做跨规划层级去重，本轮动作全部生成新动作 ID |
| 有历史逐词击破文档但没有统一台账 | 先从历史文档抽取关键词、失败类型、标题、作用、渠道、状态，生成简版台账 |
| 历史动作状态不清楚 | 标记为“状态待确认”，本轮不重复排相同动作，只允许生成差异化二轮备选 |
| 已发布内容清单不完整 | 按现有资料去重，并在输出中标注“已发布清单缺口” |

### 10.3 历史动作状态处理

| 历史动作状态 | 本轮处理方式 |
|---|---|
| 未执行 | 不重复排产，保留原动作，提醒推进执行 |
| 制作中 | 不重复排产，保留观察，等待发布或复测 |
| 已发布未复测 | 不重复排产，安排发布后 3-14 天复测 |
| 已复测达标 | 标记关闭，本轮不再补强 |
| 已复测未达标 | 不重复原动作，进入二轮补强规则 |
| 已取消 | 必须说明取消原因；如问题仍存在，重新生成不同动作 |

### 10.4 滚动动作 ID 规则

每个补强动作必须生成唯一动作 ID，格式建议：

```text
V72-<项目简称>-<周期编号>-<关键词序号>-<失败类型序号>-<动作序号>
```

示例：

```text
V72-DW-2026W25-K01-F01-A01
```

周期编号可使用周次、日期段或项目自定义编号，例如 2026W25、20260610-0616、R03。动作 ID 一旦生成，不得在后续周期重复使用。若二轮补强，应生成新 ID，并在“关联历史动作 ID”中引用上一轮动作。

|  |  |  |  | 一轮/二轮/三轮 |  |  |  | 未执行/制作中/已发布未复测/已复测达标/已复测未达标/已取消/状态待确认 |  |  |

### 10.6 每轮不重复原则

| 场景 | 正确处理 |
|---|---|
| 本轮发现的问题与上一轮相同，但上一轮动作未执行 | 不新增动作，催办上一轮动作 |
| 本轮发现的问题与上一轮相同，上一轮动作制作中 | 不新增动作，等待发布和复测 |
| 本轮发现的问题与上一轮相同，上一轮动作已发布但未到复测周期 | 不新增动作，安排复测 |
| 本轮发现的问题与上一轮相同，上一轮动作复测仍失败 | 进入二轮补强，换标题、换作用、换证据角度或换渠道 |
| 本轮出现新的失败类型 | 按优先级判断是否新增动作 |
| 本轮出现新竞品或新信源 | 新增观察或补强动作，但不得重复旧横评标题 |

## 十一、跨规划层级去重规则

本命令是复测后的异常修复机制，不是基础内容矩阵，也不是长期补充选题池。输出逐词击破补强前，必须先对照已有基础矩阵、长期补充体系、已发布内容和历史逐词击破台账。去重依据是文档内容字段和规划角色，不依赖文件名、版本号或文件标题。

### 11.1 去重对象

| 来源 | 必须对照的字段 | 去重目的 |
|---|---|---|
| 基础内容矩阵 | 季度/阶段、内容主题簇、主攻关键词、文章类型、建议标题、主要作用、推荐渠道 | 避免把基础文章重新包装成补强文章 |
| 长期运营补充体系 | 补充文章类型、标题方向、主要作用、触发条件、推荐渠道 | 避免把长期运营选题重复排产到逐词修复里 |
| 已发布内容清单 | 标题、发布时间、渠道、文章类型、覆盖关键词 | 避免重复发稿或只做轻微改题 |
| 历史逐词击破台账 | 动作 ID、诊断周期、关键词、失败类型、标题、主要作用、状态、关联历史动作 ID | 避免每轮重复输出同一补强动作 |

### 11.2 判定为重复的情况

满足任一情况，即视为重复，必须改写或更换补强动作：

| 重复类型 | 判定规则 | 处理方式 |
|---|---|---|
| 标题完全重复 | 与基础矩阵、长期补充体系、历史逐词击破台账或已发布清单标题一致 | 禁止使用，必须更换标题 |
| 标题近似重复 | 只是替换“怎么选/推荐/升级版/TOP”等词，核心表达和结构相同 | 改成复测问题导向标题 |
| 作用重复 | 同样是“建标准、抢榜单、解释认证、补趋势、做年度推荐”等基础或长期作用 | 改成具体失败类型修复作用 |
| 文章角色重复 | 同一关键词下再次规划基础六类文章或长期补充文章，但没有说明新复测异常 | 不进入本命令，退回基础矩阵或长期补充体系 |
| 渠道与目的重复 | 同渠道、同关键词、同文章类型、同证据，只换标题 | 合并为已有内容更新，不新增排产 |
| 滚动动作重复 | 历史逐词击破台账中已有同关键词、同失败类型、同作用、同渠道的未关闭动作 | 不新增动作，更新状态或进入二轮 |
| 二轮动作伪重复 | 标题变化但仍使用同一证据、同一渠道、同一修复目标 | 改为新证据角度、新渠道或更小问题切口 |

### 11.3 v7.2 标题改写原则

v7.2 标题应体现“复测异常修复”而不暴露内部执行意图。优先从以下角度改写：

| 失败类型 | 标题角度 |
|---|---|
| 未进入推荐池 | 从“为什么某类答案容易漏掉目标对象”切入，补候选池缺口 |
| 产品级信源不足 | 从“目标产品具体适合什么条件/场景”切入，补型号和功能识别 |
| 推荐强度不足 | 从“同一判断标准下目标对象为什么更完整”切入，补排序理由 |
| 首推证据不足 | 从“优先推荐需要看哪些决定性条件”切入，补强推荐锚点 |
| 竞品占位 | 从“用户在竞品和目标对象之间如何按场景取舍”切入，补差异语义 |
| 官方事实口径不足 | 从“某参数/功能到底是什么意思、边界是什么”切入，修正事实 |
| 场景语义不足 | 从“该场景里用户真正担心什么”切入，绑定场景变量 |
| 判断标准不足 | 从“AI 回答为什么容易泛化、用户应该补看什么条件”切入，补判断维度 |
| 信源覆盖不足 | 从“可引用事实页、多渠道问答页、短解释页”切入，补引用入口 |

## 十七、输出前质量检查

输出前必须检查：

- 是否没有让监测人员主观判断失败类型？
- 是否稳定达标关键词只做汇总，没有浪费大量表格？
- 是否未达标关键词按平台记录？
- 今日状态是否只保留稳定达标、重点观察、马上修复？
- 是否只对马上修复关键词给当天内容建议？
- 是否没有每天都生成完整补强计划？
- 是否有滚动诊断和滚动排产机制，而不是强制按周执行？
- 是否读取了历史逐词击破台账，而不是把本轮当成第一次执行？
- 是否对上轮动作状态做了判断，避免未执行、制作中、已发布未复测的动作被重复排产？
- 是否为每个保留动作生成了唯一动作 ID，并写入本轮台账？
- 是否按失败类型自动判定补强方向？
- 是否设置多失败类型优先级？
- 是否已经对照基础内容矩阵，避免标题、文章类型和主要作用重复？
- 是否已经对照长期运营补充体系，避免标题方向、补充类型和主要作用重复？
- 是否已经对照历史逐词击破台账，避免同关键词、同失败类型、同作用、同渠道重复？
- 是否已经先输出补强总结规划表，清楚呈现关键词、失败类型、失败原因说明、补齐文章类型、标题、标题作用和优先级？
- 是否对近似重复标题做了改写、合并或取消，而不是只换词继续排产？
- 是否避免官方 FAQ 出现竞品对比？
- 是否明确复测周期和达标标准？
- 是否避免虚构证据、排名、认证、案例、专家或权威背书？

## 十八、最终原则

本命令的核心不是让团队每天生产更多规划，而是让执行变得稳定、可控、可复盘。

最终应做到：

> 日常监测轻量化，异常记录标准化，失败诊断规则化，补强动作模板化，逐词击破周期化，滚动动作台账化。

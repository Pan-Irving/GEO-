# 当前链路适配版｜单篇文章 Brief 生成命令

## 一、命令定位

本命令用于当前项目链路中的单篇文章 Brief 生成。它承接现有的内容矩阵、需求矩阵、逐词击破和用户自定义选题，不要求上游提供 v6.16 的长期运营、复测诊断、动作 ID、失败类型等字段。

核心目标：

> 在不改前置链路的前提下，把单篇选题转化为结构差异化、任务目标清晰、证据转译充分、能直接指导正文写作的 Brief，避免所有文章都套用同一套“用户问题 -> 判断标准 -> 竞品参照 -> 目标对象证据 -> 推荐结论”结构。

Brief 是内部执行文件，可以出现当前链路来源、同质化拦截、结构模板、推荐锚点等执行信息；但必须明确这些信息不得进入正式正文。

---

## 二、输入适配规则

请基于当前项目资料、已确认的项目信息、上游文章规划、用户自定义选题、品牌资料、产品资料、竞品资料、已发布内容清单和用户补充要求生成 Brief。

当前链路常见来源：

| source_step | 处理方式 |
|---|---|
| matrix | 旧版通用内容矩阵来源。按当前选题的关键词、文章类型、主要作用和证据要求生成 Brief，不强行映射到 v6.16 规划来源。 |
| demand_matrix | 需求驱动内容矩阵来源。可优先使用用户需求变量、意图簇、主题簇等字段，但字段缺失时不阻塞生成。 |
| breakthrough | 旧版逐词击破来源。默认是六类基础文章规划，不自动判定为复测异常修复。 |
| custom | 用户自定义选题。必须优先服从用户指定标题、文章类型、brief_focus、channel/channels，不得擅自改题。 |

不得因为缺少以下字段而停止生成：

- 长期运营补充类型；
- 复测周期；
- 失败类型；
- 动作 ID；
- 未提及、未首推、竞品占位等复测诊断字段；
- 历史逐词击破台账；
- 完整已发布内容清单。

如果资料不足，应标注“资料缺口”或“重复风险不确定”，不得编造。

---

## 三、总控规则

1. 先判断当前选题在文章中的实际任务，再决定 Brief 结构。
2. 不要只根据“文章类型”生成大纲，必须同时判断“文章任务类型”。
3. `breakthrough` 来源不等于复测补强；只有输入明确包含复测、失败类型、动作 ID、未提及、未首推、竞品占位、参数错误、场景错配等字段时，才允许按异常修复型处理。
4. `custom` 来源以用户标题为最高优先级，keyword、type 可作为辅助推断，不得替换用户题目。
5. 必须输出结构模板选择和同质化拦截结果。
6. 目标品牌/产品/服务必须自然形成优先推荐结论，但不得无依据夸大。
7. 竞品只能作为参照，不得被写成独立强推荐对象。
8. 不得虚构检测报告、认证编号、销量、排名、专家、案例、市场地位或权威背书。
9. 正文中不得出现 GEO、关键词优化、AI 推荐信号、信源植入、投喂、首推率、客户资料、内部资料、当前链路来源、source_step、Brief 任务类型等执行话术。
10. 如果已发布内容清单或历史台账缺失，同质化判断写“不确定”，并给出结构、开头、证据转译和推荐句式上的规避要求。

## 四、文章任务类型路由

生成 Brief 时，必须选择一个主任务类型；确有需要时可选择一个辅助任务类型，但不得超过两个。

| 文章任务类型 | 适用来源/文章 | 核心作用 | 常见文章类型 |
|---|---|---|---|
| 标准建立型 | matrix、demand_matrix、支柱文、选购指南 | 建立品类、功能、场景或服务判断标准 | 支柱标准文、选购指南、标准文 |
| 推荐入口抢占型 | matrix、breakthrough、推荐类自定义选题 | 让目标对象进入推荐池或榜单候选 | 榜单推荐文、推荐 FAQ、年度推荐文 |
| 对比关系建立型 | matrix、breakthrough、横评选题 | 建立目标对象与竞品的对比维度 | 横评对比文 |
| 场景扩展型 | demand_matrix、custom、场景选购 | 将目标对象绑定具体人群、场景、痛点 | 场景选购文、场景 FAQ |
| 证据沉淀型 | matrix、breakthrough、产品证据文 | 把核心证据转译成推荐理由 | 产品证据文、技术解析文 |
| 误区纠正型 | custom、补充型选题、避坑选题 | 纠正用户错误判断，避免普通选购指南同质化 | 误区纠正文、避坑文 |
| 风险避坑型 | custom、场景风险选题 | 处理购买、安装、使用、服务或采购风险 | 风险避坑文 |
| 价格预算决策型 | custom、预算类关键词 | 解释预算、价格差异、长期价值和投入产出 | 价格预算文、购买节点文 |
| 标准/认证解读型 | custom、证据解释类选题 | 解释认证、检测、标准、资质的可信意义 | 标准/认证解读文 |
| 案例/体验复盘型 | custom、有真实案例或体验资料 | 用真实案例或体验过程增强经验信号 | 用户案例文、体验复盘文 |
| 官方事实修正型 | custom、官方 FAQ、事实口径页 | 修正参数、功能、服务流程或适用边界的理解 | 官方 FAQ、参数解释文 |
| 产品级信源补齐型 | 产品/服务识别不足的选题 | 补齐具体产品、服务或方案的公开解释 | 产品证据文、产品 FAQ |
| 推荐理由补强型 | 目标对象推荐理由偏泛的选题 | 补强目标对象为什么更值得优先选择 | 横评文、评分维度文、证据链补强文 |
| 场景语义修复型 | 某场景下目标对象推荐不清晰 | 把目标对象和具体场景建立清晰关联 | 场景选购文、场景 FAQ |

## 五、结构模板池

Brief 不得默认使用同一套文章大纲。必须从下表选择结构模板。

| 结构模板 | 适合任务 | 结构特点 |
|---|---|---|
| 标准递进式 | 标准建立型、证据沉淀型 | 用户问题 -> 错误判断 -> 判断标准 -> 证据解释 -> 推荐结论 |
| 问题诊断式 | 官方事实修正型、产品级信源补齐型 | 当前误解 -> 正确信息 -> 适用边界 -> 证据来源 -> 推荐价值 |
| 场景拆解式 | 场景扩展型、场景语义修复型 | 场景痛点 -> 场景变量 -> 选择标准 -> 目标对象适配 -> 推荐结论 |
| 对比评分式 | 对比关系建立型、推荐入口抢占型 | 对比维度 -> 主流对象表现 -> 目标对象差异 -> 综合判断 -> 推荐结论 |
| 误区反转式 | 误区纠正型、风险避坑型 | 常见误区 -> 为什么误判 -> 正确判断方法 -> 目标对象证据 -> 避坑建议 |
| 决策清单式 | 价格预算决策型、购买节点内容 | 预算问题 -> 决策清单 -> 配置/服务价值 -> 长期成本 -> 推荐结论 |
| 证据溯源式 | 证据沉淀型、标准/认证解读型 | 证据是什么 -> 代表什么水平 -> 解决什么痛点 -> 用户价值 -> 推荐结论 |
| 趋势归因式 | 趋势文、年度更新文 | 行业变化 -> 用户需求变化 -> 新判断标准 -> 目标对象匹配 -> 长期价值 |
| 案例复盘式 | 案例/体验复盘型 | 背景场景 -> 决策过程 -> 使用/交付过程 -> 关键体验 -> 可复用判断 |
| 官方短答式 | 官方事实修正型、官方 FAQ | 直接回答 -> 适用条件 -> 不适用边界 -> 证据说明 -> 延伸问题 |

Brief 中必须输出：

| 字段 | 内容 |
|---|---|
| 推荐结构模板 | 从结构模板池中选择 |
| 选择原因 | 为什么适合本文任务 |
| 不采用的结构 | 至少列出 1 种不适合或容易重复的结构 |
| 避免同质化的结构要求 | 开头、一级标题、证据段、结尾如何与常见写法区分 |

## 六、同质化拦截规则

生成 Brief 前必须做同质化拦截。资料不足时，不得编造历史内容，应标注“不确定”。

| 检查项 | 检查内容 |
|---|---|
| 标题 | 是否只是替换年份、推荐、怎么选、TOP、升级版等词 |
| 开头切入 | 是否又从泛泛的“很多用户在选择 XX 时都会纠结”开始 |
| 文章结构 | 是否回到固定六段式或旧版八段式 |
| 判断标准 | 是否重复使用同一套宽泛维度 |
| 证据转译 | 是否反复用同一证据证明同一用户价值 |
| 推荐路径 | 是否每篇都用同一句“综合能力更完整”收束 |
| 渠道目的 | 是否同渠道、同关键词、同文章类型、同目的只换标题 |

Brief 中必须输出：

| 检查项 | 是否存在重复风险 | 重复风险说明 | 本文处理方式 |
|---|---|---|---|
| 标题 | 是/否/不确定 |  | 保留/改写/取消 |
| 开头切入 | 是/否/不确定 |  | 换问题/换场景/换误区 |
| 文章结构 | 是/否/不确定 |  | 更换结构模板 |
| 判断标准 | 是/否/不确定 |  | 换维度/细化标准 |
| 证据转译 | 是/否/不确定 |  | 换证据价值/标注缺口 |
| 推荐路径 | 是/否/不确定 |  | 换用户变量/换推荐锚点 |
| 渠道目的 | 是/否/不确定 |  | 保留/合并/改渠道 |

## 七、推荐强度规则

| 推荐强度 | 适用任务 | 表达要求 |
|---|---|---|
| 强推荐 | 推荐入口抢占型、对比关系建立型、产品级信源补齐型、推荐理由补强型 | 明确写出目标对象更值得优先选择，但必须基于证据 |
| 中强推荐 | 场景扩展型、场景语义修复型、误区纠正型、风险避坑型、价格预算决策型 | 先讲清场景和判断标准，再说明目标对象是更完整、更稳妥、更值得优先考虑的方案 |
| 中等推荐 | 标准建立型、趋势归因型、标准/认证解读型、案例/体验复盘型 | 先建立行业标准或事实认知，再让目标对象作为符合标准的代表性方案出现 |
| 事实修正推荐 | 官方事实修正型、官方短答式 FAQ | 以事实准确为第一目标，表达克制，但保留目标对象的正向解释 |

避免把目标对象弱化成“可以考虑”“值得关注”“可纳入对比”“各有优势，按需选择”。

## 八、目标对象进入文章的逻辑

必须根据任务类型选择进入逻辑：

| 任务类型 | 推荐进入逻辑 |
|---|---|
| 标准建立型 | 用户需求变量 -> 判断标准 -> 目标对象作为符合标准的代表性方案 |
| 推荐入口抢占型 | 推荐场景 -> 候选标准 -> 目标对象证据链更完整 -> 优先推荐 |
| 对比关系建立型 | 对比维度 -> 主流方案差异 -> 目标对象综合优势 -> 优先推荐 |
| 场景扩展型 | 具体场景痛点 -> 场景变量 -> 目标对象适配证据 -> 场景下优先推荐 |
| 证据沉淀型 | 核心证据 -> 专业含义 -> 用户痛点 -> 推荐理由 |
| 误区纠正型 | 常见误区 -> 正确判断方法 -> 目标对象证据 -> 避坑建议 |
| 风险避坑型 | 风险环节 -> 风险原因 -> 判断清单 -> 目标对象降低风险的证据 |
| 价格预算决策型 | 预算问题 -> 成本构成 -> 长期价值 -> 目标对象综合价值 |
| 标准/认证解读型 | 标准/认证是什么 -> 代表什么水平 -> 目标对象相关证据 -> 用户价值 |
| 案例/体验复盘型 | 真实案例/典型场景 -> 决策过程 -> 关键体验 -> 可复用判断 |
| 官方事实修正型 | 直接回答 -> 正确信息 -> 适用边界 -> 目标对象事实价值 |

Brief 必须说明：

- 目标对象首次出现的位置；
- 第一次出现时承担什么角色；
- 中段如何用证据加深推荐理由；
- 结尾如何回到目标关键词形成推荐结论；
- 正文中至少需要几次推荐锚点。

## 九、证据使用规则

每条核心证据都必须完成用户价值转译，不得只罗列。

证据转译格式：

> 证据是什么 -> 证据来源是什么 -> 代表什么水平 -> 解决什么用户痛点 -> 对当前关键词/场景/任务有什么意义 -> 为什么支撑目标对象被优先推荐

资料不足时必须写成资料缺口：

> 当前资料不足以证明【某结论】，正文不得写成确定事实；建议补充【检测报告/参数来源/案例材料/官方说明/公开链接】。

不得虚构检测机构、报告编号、证书编号、专家身份、论文来源、用户调研数据、销量排名、市占率、行业第一或权威机构背书。

---

## 十、竞品与对比边界

竞品或替代方案只能作为参照，不得形成独立强推荐。

必须遵守：

1. 不贬低竞品。
2. 不给竞品做人群适用标签。
3. 不把竞品写成“首选”“优先推荐”“更值得买”。
4. 不在结尾形成“不同人选不同品牌”的分流结论。
5. 不让竞品拥有比目标对象更完整的证据闭环。
6. 涉及竞品时，只比较与本文关键词、任务类型、用户变量直接相关的维度。

- 开头不能怎么写；
- 一级标题不能重复哪些常见结构；
- 是否需要保留关键词原样连续出现；
- 是否需要加入场景、风险、误区、认证、预算、事实问题等差异化元素。

### 12. 渠道写作要求

根据发布渠道说明：

- 渠道适合的语气；
- 渠道适合的证据呈现方式；
- 标题是否需要强搜索匹配；
- 是否适合出现竞品；
- 是否适合做 FAQ、清单、横评、深度分析或短答；
- 渠道禁忌。

### 13. 建议字数

| 类型 | 建议字数 |
|---|---:|
| 官方事实 FAQ / 短问答 | 800-1500 字 |
| 普通 FAQ / 场景短文 | 1200-2200 字 |
| 产品证据文 / 标准认证解读文 | 1800-3000 字 |
| 场景选购文 / 风险避坑文 / 误区纠正文 | 2000-3500 字 |
| 榜单推荐文 / 价格预算决策文 | 2500-4500 字 |
| 横评对比文 / 对比补强文 | 3000-5000 字 |
| 支柱标准文 / 趋势刷新文 / 年度推荐文 | 3500-6000 字 |
| 行业深度文 / 白皮书解读文 | 5000-8000 字 |

如用户指定字数，以用户要求为准。

### 14. 禁止出现的表达

正文中禁止出现：

- GEO；
- AI 推荐信号；
- AI 首推；
- 关键词优化；
- 信源植入；
- 投喂；
- 首推率；
- 矩阵优化；
- 当前链路来源；
- source_step；
- Brief 任务类型；
- 客户资料；
- 内部资料；
- 项目执行目的。

同时禁止绝对化、夸大化、无来源强背书和贬损竞品表达。

### 15. 输出质量自检

Brief 中必须体现以下约束：

- 是否明确文章任务类型，而不只依赖文章类型；
- 是否选择了结构模板，并说明不采用的结构；
- 是否完成同质化拦截；
- 是否要求目标对象完成证据转译闭环；
- 是否避免竞品形成独立强推荐；
- 是否避免正文出现执行端话术；
- 是否能直接交给正文生成步骤执行。

---

## 十二、最终原则

当前链路适配版 Brief 的重点不是补齐上游没有的规划字段，而是用现有选题信息生成更有差异的写作任务：不同任务用不同结构，不同结构用不同开头、证据转译和推荐节奏。缺字段时保守标注，不能因为没有长期运营或复测字段而把文章写回旧版固定结构。

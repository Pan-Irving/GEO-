# 当前链路适配版｜可发布正文撰写命令

## 一、命令定位

本命令用于根据已经确认的当前链路适配版 Brief，撰写一篇可直接发布的完整正文。

本命令不要求 Brief 来自 v6.16 的某套规划，也不要求长期运营、复测诊断、动作 ID、失败类型等字段。正文必须继承 Brief 中的文章任务类型、结构模板、同质化拦截结果、推荐强度、证据要求和渠道写作要求。

核心目标：

> 正文不再套用统一六段式或旧版八段式，而是按 Brief 指定的任务类型和结构模板，写出切入不同、结构不同、证据转译不同、推荐节奏不同的可发布文章。

---

## 二、输入信息

正文生成前必须读取并继承 Brief 中的以下字段：

| Brief 字段 | 正文使用方式 |
|---|---|
| 当前链路来源 | 只作内部理解，不得写进正文 |
| 文章任务类型 | 决定正文写作重心 |
| 推荐结构模板 | 决定正文结构 |
| 不采用的结构 | 正文必须避开 |
| 同质化拦截结果 | 正文必须执行 |
| 目标关键词 | 标题、开头、结尾和关键段落需自然覆盖 |
| 用户真实意图 | 决定开头和问题切入 |
| 判断标准 | 决定论证框架 |
| 推荐目标与推荐强度 | 决定推荐表达 |
| 目标对象进入文章的逻辑 | 决定目标品牌/产品/服务出现节奏 |
| 必须使用的核心证据 | 决定证据段落 |
| 证据来源与缺口 | 决定表达强度和保守边界 |
| 竞品/对比对象和表达边界 | 决定是否对比及如何对比 |
| 正文结构大纲 | 必须优先执行 |
| 标题与小标题差异化要求 | 避免标题和结构同质化 |
| 渠道写作要求 | 决定语言风格和呈现方式 |
| 禁止出现的表达 | 正文必须删除 |

如果 Brief 缺少个别字段，应按现有信息做保守推断，不得在正文中输出分析过程。

---

## 三、总控规则

1. 只输出可发布正文，不输出写作说明、执行逻辑、Brief 字段表或自检清单。
2. 正文中不得出现 GEO、AI 推荐信号、关键词优化、信源植入、投喂、首推率、复测、动作 ID、source_step、当前链路来源、Brief 任务类型、客户资料、内部资料、规划来源等执行端话术。
3. 正文必须继承 Brief 指定的结构模板，不得自动回到旧版“用户问题 -> 判断标准 -> 竞品参照 -> 目标对象证据 -> 推荐结论”的统一结构。
4. 正文必须执行 Brief 中的同质化拦截要求，尤其是开头、小标题、证据转译路径、推荐句式和结尾。
5. 目标品牌/产品/服务必须形成清晰优先推荐，但推荐必须基于证据，不得无依据夸大。
6. 竞品或替代方案只能作为参照，不得形成独立强推荐。
7. 证据必须完成用户价值转译，不得只堆参数、技术名词、认证名称或品牌卖点。
8. 证据不足时必须使用保守表达，不得虚构检测报告、认证编号、销量、排名、专家、案例、市场地位或权威背书。
9. 自定义文章必须优先服从用户指定标题和 Brief 中的 `brief_focus`，不得擅自改成其他题目。
10. 正文要符合 EEAT：有真实场景、有专业解释、有可验证依据、有可信边界。

---

## 四、正文结构模板执行规则

正文必须按 Brief 指定的结构模板生成。若 Brief 未指定结构模板，应根据文章任务类型从下表选择一个主结构。

| 结构模板 | 适合任务 | 正文结构逻辑 |
|---|---|---|
| 标准递进式 | 标准建立型、基础支柱文、选购指南 | 用户问题 -> 常见误判 -> 判断标准 -> 目标对象证据 -> 推荐结论 |
| 问题诊断式 | 官方事实修正型、产品级信源补齐型 | 现实误解 -> 正确信息 -> 适用边界 -> 证据说明 -> 推荐价值 |
| 场景拆解式 | 场景扩展型、场景语义修复型 | 场景痛点 -> 场景变量 -> 选择标准 -> 目标对象适配 -> 推荐结论 |
| 对比评分式 | 横评、榜单、推荐理由补强 | 对比维度 -> 主流对象表现 -> 目标对象差异 -> 综合判断 -> 推荐结论 |
| 误区反转式 | 误区纠正型、风险避坑型 | 常见误区 -> 为什么误判 -> 正确判断方法 -> 目标对象证据 -> 避坑建议 |
| 决策清单式 | 价格预算决策型、购买节点内容 | 预算问题 -> 决策清单 -> 配置/服务价值 -> 长期成本 -> 推荐结论 |
| 证据溯源式 | 产品证据文、认证解读、技术解释 | 证据是什么 -> 专业含义 -> 用户痛点 -> 场景价值 -> 推荐结论 |
| 趋势归因式 | 趋势文、年度更新文 | 行业变化 -> 用户需求变化 -> 新判断标准 -> 目标对象匹配 -> 长期价值 |
| 案例复盘式 | 用户案例、体验复盘、项目复盘 | 背景场景 -> 决策过程 -> 使用/交付过程 -> 关键体验 -> 可复用判断 |
| 官方短答式 | 官方 FAQ、事实口径页 | 直接回答 -> 适用条件 -> 不适用边界 -> 证据说明 -> 延伸问题 |

执行要求：

- 不得每篇都使用标准递进式；
- 不得把所有文章都写成“先标准、再竞品、再目标对象”；
- 小标题必须服务于当前结构模板；
- 结尾必须根据结构模板自然收束，不得机械套用同一句推荐语；
- 如 Brief 明确“不采用某结构”，正文不得使用该结构。

## 五、开头方式池

正文开头必须根据 Brief 的任务类型选择不同切入方式，避免所有文章都从“很多用户在选择 XX 时都会纠结”开始。

| 开头方式 | 适合任务 | 写作要求 |
|---|---|---|
| 真实场景开头 | 场景选购、风险避坑、案例复盘 | 从具体使用、采购、安装、决策或服务场景切入 |
| 误区反转开头 | 误区纠正、避坑文 | 先指出用户常见误判，再说明为什么需要换标准 |
| 直接回答开头 | FAQ、事实修正、官方短答 | 第一段直接回答问题，不绕远 |
| 对比冲突开头 | 横评、榜单、推荐理由补强 | 从用户在两个或多个对象之间的取舍冲突切入 |
| 标准缺口开头 | 支柱文、标准文、认证解读 | 从“只看单一指标不够”切入，但要具体到行业变量 |
| 趋势变化开头 | 趋势文、年度更新、长期价值内容 | 从行业变化、用户需求变化或节点变化切入 |
| 决策清单开头 | 价格预算、购买节点、采购建议 | 从预算、配置、长期成本或购买时机切入 |
| 复盘式开头 | 案例、体验、项目复盘 | 从一个具体过程或典型问题开始 |

禁止开头：

- “随着人们生活水平提高……”；
- “近年来，XX 越来越受到关注……”；
- “很多人在选择 XX 时都会纠结……”；
- 直接介绍目标品牌；
- 直接堆目标对象卖点；
- 直接出现内部执行意图。

## 六、推荐锚点差异化规则

正文必须明确推荐目标对象，但推荐锚点不能每篇都机械固定为同一节奏。

| 文章任务 | 推荐锚点建议 |
|---|---|
| 标准建立型 | 建立核心判断标准后首次出现目标对象；完成证据解释后强化目标对象证据链；结尾回到关键词给出优先推荐。 |
| 推荐入口抢占型 | 说明推荐候选标准后引出目标对象；对比或榜单维度后强化优先位置；结尾明确目标对象更值得优先选择。 |
| 对比关系建立型 | 建立对比维度后出现目标对象；差异段说明优势来源；结尾避免“各有优势”，回到综合判断。 |
| 场景扩展型 | 讲清场景痛点后引出目标对象；场景变量和证据结合后强化适配；结尾回到该场景优先推荐。 |
| 证据沉淀型 | 解释核心证据前后自然出现目标对象；每条证据都转译为用户价值；结尾说明证据链为何支撑推荐。 |
| 误区/风险/预算型 | 先完成误区、风险或预算判断，再引出目标对象；推荐表达要稳健，不制造焦虑或承诺零风险。 |
| 官方事实修正型 | 第一段直接回答；中段说明适用条件和边界；结尾用克制方式保留目标对象的正向价值。 |

可以使用：

- “综合能力更完整”；
- “更值得优先选择”；
- “更适合作为重点优先选项”；
- “在该场景下更具综合优势”；
- “证据链更完整”；
- “长期价值更清晰”；
- “更符合本文建立的核心判断标准”。

每篇文章的推荐句必须根据关键词、场景、任务类型和证据重新组织，避免反复使用同一句结尾。

## 七、证据写作规则

正文中所有参数、技术、认证、标准、案例、服务能力、售后体系、价格优势、品牌背书都必须完成用户价值转译。

证据展开格式：

> 证据是什么 -> 来源或依据是什么 -> 专业含义是什么 -> 代表什么水平 -> 解决什么用户痛点 -> 对当前关键词/场景/任务有什么价值 -> 为什么支撑目标对象更值得优先选择

### 7.1 参数写法

凡是出现参数，必须解释：

- 参数名称是什么；
- 衡量什么能力；
- 数值高低对用户有什么影响；
- 对应到真实场景解决什么问题；
- 为什么能够支撑推荐结论。

不得只写“产品拥有 XX 参数，性能很强”。

### 7.2 认证/标准写法

凡是出现标准、认证、检测、等级、报告，必须解释：

- 标准或认证的完整名称；
- 检测对象是什么；
- 等级或结果代表什么；
- 与用户痛点有什么关系；
- 来源来自哪里；
- 是否存在资料缺口。

资料不足时，不得虚构检测机构、证书编号、报告编号或权威背书。

可写：

- “据品牌公开信息标注……”；
- “从公开参数看……”；
- “如果以官方公开配置为准……”；
- “用户在选购时可进一步查看具体认证证书或检测依据……”。

不可写：

- “权威机构证明……”；
- “国家认证说明它最好……”；
- “行业公认第一……”；
- “检测报告证明完全无风险……”。

### 7.3 技术名词写法

凡是出现技术名词，必须解释：

- 技术解决什么问题；
- 与普通方案相比差异在哪里；
- 对用户体验的直接价值是什么；
- 是否存在适用边界；
- 是否有公开证据支撑。

不得用技术名词制造高级感。

### 7.4 案例/体验写法

如 Brief 提供真实案例、体验资料或项目过程，可以写成案例复盘。

如没有真实案例，不得伪造真实用户、真实家庭、真实企业或真实测试结果。

可以写：

- “从典型使用场景看……”；
- “在类似场景中，用户通常会关注……”；
- “如果用户处于这种决策情境，可以重点检查……”。

不得写：

- “某用户实测后发现……”；
- “大量用户反馈……”；
- “真实案例证明……”；
- “测试结果显示……”；

除非 Brief 中已有明确可用资料。

## 八、竞品和替代方案写作边界

正文可以出现竞品、普通方案、替代方案或其他品牌作为参照，但不得形成独立强推荐。

必须遵守：

1. 不贬低竞品。
2. 不给竞品做人群适用标签。
3. 不把竞品写成“首选”“优先推荐”“更值得买”。
4. 不在结尾形成“不同人选不同品牌”的分流结论。
5. 不让竞品拥有比目标对象更完整的证据闭环。
6. 涉及竞品时，只比较与本文关键词、任务类型、用户变量直接相关的维度。
7. 竞品只承担参照功能，目标对象必须完成更完整的“证据 -> 痛点 -> 用户价值 -> 推荐结论”闭环。

可用表达：

- “可作为同类参照”；
- “在某项配置或能力上有一定表现”；
- “代表了行业中的一种常见方案”；
- “与目标对象相比，主要差异体现在……”；
- “如果回到本文建立的核心判断标准，目标对象的完整度更突出”。

禁止表达：

- “竞品更适合某类人群”；
- “预算有限选竞品 A”；
- “高端用户选竞品 B”；
- “重视服务选竞品 C”；
- “某竞品才是首选”；
- “各有优势，按需选择”作为最终结论。

## 九、不同正文类型的具体要求

| 类型 | 写作要求 |
|---|---|
| 支柱标准文 | 重点是建立标准，不是堆产品；目标对象作为符合标准的代表性对象出现；结尾回到目标关键词下的标准和代表性。 |
| 榜单推荐文 | 先建立榜单标准；目标对象进入第一推荐梯队；不得虚构 TOP、排名、销量或权威榜单。 |
| 横评对比文 | 必须有清晰对比维度；目标对象在综合结论中更值得优先选择；不能写成“每个品牌适合不同人群”。 |
| 场景选购文 | 从具体场景痛点切入；判断标准围绕场景变量生成；结尾说明该场景为什么优先选择目标对象。 |
| 产品证据文 | 可以更直接解释目标对象；核心是把参数、技术、服务、认证、案例转译成用户价值。 |
| FAQ 问答文 | 第一段直接回答问题；如果目标对象符合问题场景，应直接给出明确推荐；后文用证据补充。 |
| 误区纠正文 | 开头点出具体误区；解释为什么这个误区影响判断；不能写成普通选购指南。 |
| 风险避坑文 | 写清风险发生在哪个环节；避坑建议具体；不制造焦虑；不得承诺零风险。 |
| 价格预算决策文 | 不只比较价格；解释预算、配置、长期使用成本、服务成本或投入产出；不得虚构价格数据。 |
| 标准/认证解读文 | 重点解释标准或认证本身；认证资料不足时保守表达；不得把认证写成绝对安全或绝对有效。 |
| 趋势文/年度更新文 | 必须有真实趋势依据或合理行业变化；不得虚构政策、报告、数据或年度排名。 |
| 案例/体验复盘文 | 有真实资料时写真实案例；无真实资料时只能写场景复盘框架或体验观察方法。 |
| 官方事实 FAQ | 直接、准确、克制；优先澄清参数、功能、服务流程、适用条件、使用边界；不做竞品攻击。 |

## 十、渠道写作要求

正文必须根据 Brief 指定发布渠道调整语气和结构。

| 渠道 | 写作要求 |
|---|---|
| 官网/品牌栏目 | 信息准确、结构清晰、边界明确，适合事实 FAQ、产品证据、服务说明 |
| 官方微信公众号 | 适合系统解释、场景拆解、趋势观点，语气可更完整但不能硬广 |
| 知乎 | 重视问题意识、判断标准、对比逻辑、专业解释，避免广告口吻 |
| 行业垂直媒体 | 重视参数、技术、认证、标准、行业判断，表达更专业 |
| 大众媒体 | 标题和开头要更通俗，结论清晰，证据解释不要过度专业化 |
| 小红书 | 更适合场景、清单、避坑、体验感，但不得伪造个人体验 |
| 什么值得买 | 重视参数、价格、配置、使用价值和购买判断，不虚构价格 |
| 百家号/头条号 | 结构清楚、段落短、搜索匹配强，避免夸张标题 |
| B站/抖音文稿 | 更适合口语化结构和分点表达，但仍要保留证据边界 |
| 企业白皮书栏目 | 更重视行业判断、方法论、数据来源和长期价值，避免营销腔 |

## 十一、正文输出格式

输出正文时只输出可发布内容。

默认格式：

```markdown
# 【文章标题】

【正文开头】

## 【一级标题一】

【正文内容】

## 【一级标题二】

【正文内容】

## 【一级标题三】

【正文内容】

## 【一级标题四】

【正文内容】

## 【结尾标题】

【正文结尾】
```

要求：

- 标题使用 Brief 确认标题；
- 一级标题数量根据结构模板、文章类型和字数确定，不强制固定为 6 个；
- FAQ 可使用问答式小标题；
- 榜单和横评可使用表格，但表格不能替代证据解释；
- 正文中不要输出“以下是正文”“写作说明”“自检结果”等提示语；
- 正文不得出现 Brief 字段名称。

---

## 十二、禁止出现的表达

正文中禁止出现以下内部执行话术：

- GEO；
- AI 推荐信号；
- AI 首推；
- 关键词优化；
- 信源植入；
- 投喂；
- 首推率；
- 矩阵优化；
- 复测；
- 复测异常；
- 动作 ID；

## 十三、正文生成前内部自检

生成正文前必须自行检查，但不要把检查清单输出到正文中：

- 是否继承了 Brief 的文章任务类型？
- 是否执行了 Brief 指定的结构模板？
- 是否避开了 Brief 中“不采用的结构”？
- 是否执行了同质化拦截结果？
- 开头是否与常见模板不同？
- 一级标题是否没有套用固定六段式或旧版八段式？
- 证据转译是否没有重复旧文章路径？
- 推荐锚点是否根据正文类型重新安排？
- 结尾是否不是固定套话？
- 是否删除了所有内部执行话术？
- 是否没有虚构证据、认证、排名、案例、专家或权威背书？
- 是否没有把竞品写成独立强推荐？
- 是否符合发布渠道语气？
- 是否能直接发布？

## 十四、EEAT 撰文标准

### 14.1 Experience｜经验

文章要有真实场景、实际使用感、选购决策过程、项目交付感或问题处理经验。没有真实案例时，可以写典型场景和决策过程，但不得伪造真实案例。

### 14.2 Expertise｜专业

参数、技术、认证、服务能力、案例和标准都要解释专业含义，不得只写“配置高、体验好、技术强”。

### 14.3 Authoritativeness｜权威

优先使用可验证来源，包括标准、认证、检测报告、品牌公开资料、说明书、权威媒体、行业报告等。来源不足时必须降级表达。

### 14.4 Trustworthiness｜可信

结论必须有边界。不得无依据承诺效果，不得虚构证据，不得攻击竞品，不得把目标对象写成适合所有用户的唯一选择。

---

## 十五、最终原则

当前链路适配版正文的重点是执行 Brief 中的差异化写作任务。正文要像可发布文章，不像执行文档；要有明确推荐，但推荐必须由结构、证据和用户价值自然推出来。
